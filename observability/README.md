# AEGIS Observability Module

Production-grade observability framework for distributed AI inference systems. Provides metrics collection, structured logging, distributed tracing, and health probing.

## Features

### Metrics (Prometheus)

Comprehensive metrics across all system layers:

- **Inference Metrics**
  - `aegis_inference_requests_total` - Total inference requests by backend
  - `aegis_inference_errors_total` - Total errors by backend
  - `aegis_inference_latency_ms` - Inference latency histogram (p50, p99, p99.9)
  - `aegis_inference_tokens_generated` - Tokens generated by backend

- **Backend Health Metrics**
  - `aegis_backend_health` - Health status (1.0 = healthy, 0.0 = unhealthy)
  - `aegis_backend_active_requests` - Active concurrent requests per backend
  - `aegis_backend_latency_ms` - Per-backend response latency

- **Resilience Metrics**
  - `aegis_circuit_breaker_state` - Current state (0=Closed, 1=Open, 2=HalfOpen)
  - `aegis_circuit_breaker_failures_total` - Cumulative failures
  - `aegis_circuit_breaker_opens_total` - Number of times circuit opened
  - `aegis_retry_attempts_total` - Total retry attempts
  - `aegis_retry_success_total` - Successful retries
  - `aegis_timeout_errors_total` - Timeout errors
  - `aegis_degradation_level` - Degradation level (0=Healthy, 1=Degraded, 2=Critical)
  - `aegis_degradation_fallback_uses_total` - Times fallback used

### Tracing (OpenTelemetry)

Distributed tracing with structured JSON logging:

- JSON-formatted structured logs with context
- OpenTelemetry integration for distributed tracing
- Jaeger export support for trace visualization
- Custom span creation for instrumentation
- Trace correlation across microservices

### Health Probes (Kubernetes)

Liveness and readiness probes for orchestration:

- **Liveness Probe**: Indicates if service is alive (HTTP /health/live)
- **Readiness Probe**: Indicates if service is ready for traffic (HTTP /health/ready)
- Component-level readiness: inference engine, backends, resilience layer
- Automatic uptime tracking and state management

## Usage Patterns

### Pattern 1: Metrics Recording

```rust
use observability::METRICS;

// Record inference request
METRICS.record_inference_request("vllm-1", 150.0); // 150ms latency

// Record backend health
METRICS.record_backend_health("hf-api", 1.0); // healthy

// Record errors
METRICS.record_inference_error("vllm-1");

// Record circuit breaker state
METRICS.record_circuit_breaker_state("vllm-1", 0); // 0=Closed
METRICS.record_circuit_breaker_failure("vllm-1");
```

### Pattern 2: Distributed Tracing

```rust
use observability::{init_tracing, create_span, TracingConfig};

// Initialize at startup
let config = TracingConfig {
    json_format: true,
    log_level: "debug".to_string(),
    jaeger_endpoint: Some("http://localhost:6831".to_string()),
};
init_tracing(&config);

// Create spans for operations
let span = create_span("inference_request", &[
    ("model", "llama-7b"),
    ("backend", "vllm"),
]);

// Automatic logging
tracing::info!("Processing inference request");
tracing::warn!("Retry attempt 2/3");
tracing::error!("Circuit breaker opened");
```

### Pattern 3: Health Management

```rust
use observability::HealthManager;

let health = HealthManager::new();

// Mark components ready during startup
health.mark_backends_ready();
health.mark_inference_ready();

// Check health status
let readiness = health.get_readiness();
if !readiness.ready {
    eprintln!("Service not ready: {:?}", readiness.checks);
}

let liveness = health.get_liveness();
println!("Service alive for {}s", liveness.uptime_secs);
```

## Integration Points

### With inference-backends

```rust
// In backend implementations, record metrics:
METRICS.record_inference_request("hf-api", latency_ms);
METRICS.record_backend_health("hf-api", if health { 1.0 } else { 0.0 });

// Log operations:
tracing::info!(
    backend = "hf-api",
    model = model_id,
    tokens = response.tokens.len(),
    "Inference completed"
);
```

### With resilience layer

```rust
// In circuit breaker:
METRICS.record_circuit_breaker_state(backend, state);
METRICS.record_circuit_breaker_failure(backend);

// In retry handler:
METRICS.record_retry_attempt(backend);
METRICS.record_retry_success(backend);
tracing::info!(attempt = attempt_num, "Retrying request");

// In timeout handler:
METRICS.record_timeout_error();
tracing::warn!(timeout_ms = timeout, "Request timeout");

// In degradation handler:
METRICS.record_degradation_level(level);
tracing::warn!("Service degraded, using fallback");
```

## Configuration

### TracingConfig

```yaml
# observability.yaml
tracing:
  log_level: "info"           # debug, info, warn, error
  json_format: true           # JSON vs pretty-print
  jaeger_endpoint: "http://localhost:6831"
  opentelemetry:
    enabled: true
    sample_rate: 1.0          # 0.0-1.0 sampling rate
```

### Prometheus Scrape Configuration

```yaml
# prometheus.yml
scrape_configs:
  - job_name: 'aegis'
    static_configs:
      - targets: ['localhost:8000']
    metrics_path: '/metrics'
    scrape_interval: 15s
    scrape_timeout: 10s
```

### Kubernetes Integration

```yaml
# Pod health checks

  httpGet:
    path: /health/live
    port: 8000
  initialDelaySeconds: 10
  periodSeconds: 10

readinessProbe:
  httpGet:
    path: /health/ready
    port: 8000
  initialDelaySeconds: 5
  periodSeconds: 5
```

## Alerting Rules

### Critical Alerts

- Circuit breaker open for >1 minute
- Quorum lost (healthy peers < 2)
- Error rate > 5%
- P99 latency > 10 seconds
- Service not ready for > 30 seconds

### Warning Alerts

- Error rate > 1%
- P99 latency > 5 seconds
- Backend health < 0.9
- Retry attempts high (>10/min)
- Degradation level elevated

## Performance Impact

- **Metrics**: ~0.1ms per recording (negligible)
- **Tracing**: ~0.5-1ms per span (configurable sampling)
- **Health checks**: ~1ms per probe
- **Memory**: ~50MB for metrics + tracing buffers

## Best Practices

1. **Sampling**: Enable sampling for high-volume operations (set sample_rate < 1.0)
2. **Cardinality**: Avoid unbounded label values (use fixed set of backends)
3. **Retention**: Set Prometheus retention to 15 days (cost vs historical analysis tradeoff)
4. **Alerting**: Configure alerts before production deployment
5. **Dashboards**: Create Grafana dashboards for visualization
6. **Context**: Pass trace context through request headers (W3C Trace Context)
7. **Graceful shutdown**: Flush remaining traces/metrics on shutdown

## Export Formats

- **Prometheus**: OpenMetrics format (/metrics endpoint)
- **Jaeger**: OTLP/gRPC to Jaeger collector
- **Structured Logs**: JSON lines to stdout/file
- **Health**: JSON responses from health probes

## Troubleshooting

### Metrics not appearing

Check that METRICS singleton is initialized and MetricsRegistry::new() was called during startup.

### Traces not appearing in Jaeger

Verify Jaeger collector is running and reachable. Check jaeger_endpoint configuration. Ensure trace sample_rate > 0.

### Health probe returns unhealthy

Check component readiness status: `health.get_readiness().checks` shows which components failed.
